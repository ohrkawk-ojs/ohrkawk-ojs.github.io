회사에서 새 도구를 들여올 때, 처음으로 부딪히는 것은 기능표가 아닙니다. 출입문입니다.

누가 계정을 받을 것인가. 어느 팀까지 열어줄 것인가. 개인 메일로 가입해도 되는가. 비용은 부서 예산으로 잡을 것인가, 전사 공통비로 볼 것인가. 업무 파일을 올려도 되는가. 외부 SaaS와 연결해도 되는가. 처음에는 사소한 관리 질문처럼 보이지만, 대기업 사무실에서는 이 질문들이 도구의 운명을 결정합니다. 아무리 좋은 도구라도 출입문이 없으면 공식 업무 도구가 되기 어렵습니다.

AI 에이전트는 이 문제를 더 크게 만듭니다. 단순히 문장을 생성하는 챗봇이라면 사용 범위를 좁힐 수 있습니다. 하지만 Claude Code나 Claude Desktop처럼 개발 환경과 로컬 파일, 도구 호출, 사내 시스템 연결 가능성을 품은 앱은 다릅니다. 개인 생산성 도구로 시작한 AI가 조직의 업무 흐름 안으로 들어오는 순간, 회사는 “잘 쓰게 하자”보다 먼저 “어디까지 열고, 어떻게 기록하고, 누가 책임질 것인가”를 묻게 됩니다.

그래서 AWS가 공개한 두 개의 글은 하나의 흐름으로 읽힙니다. 하나는 Claude Code와 Claude Desktop을 기업이 중앙에서 배포·관리할 수 있도록 하는 self-hosted control plane, 즉 **Claude apps gateway for AWS**입니다. 다른 하나는 MCP 도구 설계에서 실제로 마주치는 트레이드오프를 정리한 글입니다. 겉으로는 각각 배포 관리와 개발자용 프로토콜 이야기처럼 보이지만, 기업 AX 관점에서 보면 같은 질문을 향합니다. AI 에이전트가 회사 업무 도구가 되려면, 모델보다 먼저 출입문과 도구의 언어가 설계되어야 한다는 질문입니다.

## ▸ 개인 앱이 회사 도구가 되는 순간 : 권한의 문이 먼저 열린다

AWS 공식 블로그에 따르면 Claude apps gateway for AWS는 조직이 Claude Code와 Claude Desktop을 배포할 때 인증, 정책, 텔레메트리, 라우팅을 중앙에서 관리하기 위한 자체 호스팅형 게이트웨이입니다. AWS는 이 글에서 개발자별 클라우드 자격증명 관리, 설정 배포, 사용량과 비용 추적 문제가 팀 단위 확산의 부담이 된다고 설명합니다.

본문에서 눈에 띄는 것은 게이트웨이가 단순 프록시가 아니라 조직 통제 지점으로 설계되어 있다는 점입니다. 사용자는 SSO로 로그인하고, 게이트웨이는 CLI가 이후 요청에 사용할 짧은 수명의 토큰을 발급합니다. 관리자는 서버에서 허용 모델, 도구 권한, 기본 설정을 한 번 정의하고, 이를 IdP 그룹 단위로 적용할 수 있습니다. 각 요청의 사용량 메트릭은 OpenTelemetry Protocol을 통해 Amazon CloudWatch, Amazon Managed Service for Prometheus 또는 서드파티 플랫폼으로 보낼 수 있다고 설명됩니다.

이 구조는 기업이 AI 앱을 바라보는 관점이 바뀌고 있음을 보여줍니다. 처음의 질문은 “Claude가 얼마나 똑똑한가”였을 수 있습니다. 하지만 조직 배포 단계의 질문은 다릅니다. 누가 어떤 모델을 쓸 수 있는가. 어떤 도구 호출을 허용할 것인가. 사용량과 비용은 어디에 기록할 것인가. 퇴사자나 부서 이동자의 권한은 어떻게 회수할 것인가. 민감 데이터가 포함된 업무에서 어떤 정책을 강제할 것인가.

AI 에이전트가 개인의 노트북 안에 머물 때는 이런 질문을 뒤로 미룰 수 있습니다. 그러나 회사의 개발·문서·분석 업무에 연결되는 순간, 에이전트는 또 하나의 업무 시스템이 됩니다. 업무 시스템에는 늘 출입문이 필요합니다.

## ▸ 문제는 API 목록이 아니라 AI가 이해하는 도구의 언어다

같은 날 전후로 AWS가 정리한 MCP 툴 설계 글은 다른 층위의 출입문을 다룹니다. MCP는 모델과 외부 도구를 연결하는 표준 인터페이스로 주목받고 있지만, 실제 현장에서는 프로토콜을 붙였다고 바로 좋은 에이전트가 되는 것이 아닙니다. AWS는 MCP 도구가 실패하는 원인이 프로토콜 자체보다 도구 설계에 있는 경우가 많다고 설명합니다.

가장 쉬운 실수는 기존 API를 그대로 AI에게 노출하는 것입니다. 사람 개발자를 위한 API는 많은 옵션과 예외, 내부 용어를 포함합니다. 하지만 LLM은 그 API 목록을 업무 맥락 안에서 해석해야 합니다. 도구 설명이 길고 모호하면 컨텍스트를 많이 잡아먹고, 파라미터가 복잡하면 잘못된 값을 넣기 쉽고, 실패하면 재시도를 반복하면서 비용과 시간이 늘어납니다. 결국 “연결했다”는 사실보다 “AI가 올바르게 선택하고 안전하게 실행할 수 있게 설계했는가”가 더 중요해집니다.

기업 내부 API는 대개 사람과 조직의 역사를 품고 있습니다. 오래된 시스템의 필드명, 부서별 예외 규칙, 승인 단계에 따라 달라지는 입력값, 운영자가 암묵적으로 알고 있는 순서가 섞여 있습니다. 이 API를 그대로 MCP 도구로 열어놓으면 에이전트는 회사 일을 하는 것이 아니라 회사의 복잡성을 그대로 떠안게 됩니다.

그래서 MCP 시대의 실무는 단순한 연동 작업이 아닙니다. 업무 목적별로 도구를 좁히고, 이름과 설명을 LLM이 오해하지 않게 쓰고, 파라미터를 줄이고, 실패했을 때의 응답을 명확히 만들고, 권한과 감사 로그를 함께 설계하는 일입니다. 말하자면 AI에게 회사 시스템을 보여주는 문을 새로 만드는 작업입니다. 문이 너무 넓으면 위험하고, 너무 좁으면 쓸모가 없습니다.

## ▸ 통제인가 활성화인가 : 같은 게이트웨이의 두 얼굴

이 변화를 통제 강화로만 읽을 수도 있습니다. 중앙 게이트웨이가 생기면 개인이 자유롭게 쓰던 AI 도구는 조직 정책 아래 들어갑니다. 허용 모델이 제한되고, 도구 권한이 나뉘고, 사용량이 기록됩니다. 개발자 입장에서는 답답하게 느껴질 수 있습니다. 빠르게 실험하고 싶은 사람에게 중앙 정책은 마찰처럼 보입니다.

하지만 반대로 보면, 이런 통제 지점이 있어야 AI 앱은 더 넓게 퍼질 수 있습니다. 보안팀이 아무것도 볼 수 없고, 비용 담당자가 사용량을 추적할 수 없고, IT가 퇴사자 권한을 회수할 수 없다면 대기업은 AI 에이전트를 공식 도구로 열기 어렵습니다. 통제는 속도를 늦추는 장치이기도 하지만, 일정 규모 이상에서는 확산을 가능하게 하는 조건이기도 합니다.

MCP 툴 설계도 마찬가지입니다. 도구를 잘게 나누고 설명을 정리하고 권한을 붙이는 일은 초기에는 번거롭습니다. 기존 API를 그대로 붙이는 것보다 느립니다. 그러나 이 작업을 하지 않으면 에이전트의 실패는 사용자의 불신으로 돌아옵니다. 한 번 엉뚱한 데이터를 가져오고, 한 번 잘못된 티켓을 수정하고, 한 번 권한 밖 정보를 요약하면 현업은 AI를 “편한 도구”가 아니라 “감시해야 할 위험”으로 기억합니다.

결국 게이트웨이와 MCP 설계는 AI를 막기 위한 장치가 아니라, 조직 안에서 지속적으로 쓰기 위한 최소한의 운영 구조입니다. 개인의 생산성 도구가 회사의 업무 인프라가 되려면 자유만으로는 부족합니다. 자유가 반복 가능한 운영으로 바뀌는 지점에는 언제나 정책, 기록, 비용, 권한이 서 있습니다.

## ▸ 한국의 사무실에서 이 뉴스를 읽는 법

한국 대기업의 AX 프로젝트에서 이 뉴스가 중요한 이유는 분명합니다. 많은 조직이 이미 Copilot, Claude, ChatGPT Enterprise, 사내 LLM, RPA, 업무 자동화 플랫폼을 동시에 검토하고 있습니다. 그런데 현장에서 자주 막히는 지점은 모델 성능이 아닙니다. “이걸 우리 업무망에서 써도 되는가”, “어떤 문서까지 읽게 할 것인가”, “비용은 누가 볼 것인가”, “AI가 실행한 작업은 감사 로그에 남는가” 같은 질문입니다.

Claude apps gateway 같은 구조는 앞으로 기업 AI 도입의 기준선을 보여줍니다. AI 도구를 전사에 열기 전에 SSO, 그룹별 정책, 허용 모델, 도구 권한, 텔레메트리, 비용 관측성을 갖춰야 합니다. 이 기준은 특정 벤더 하나의 문제가 아닙니다. 어떤 에이전트 플랫폼을 쓰든 대기업은 결국 같은 질문으로 돌아옵니다. AI가 누구의 권한으로 움직이며, 어떤 데이터를 보고, 어떤 도구를 호출하고, 그 흔적이 어디에 남는가.

MCP 툴 설계는 업무자동화팀과 현업에게 더 직접적인 숙제를 줍니다. 사내 시스템을 AI와 연결하려면 API 문서를 던져주는 것으로 끝나지 않습니다. 구매 요청을 생성하는 도구, 계약서 초안을 검토하는 도구, 장애 티켓의 영향을 분석하는 도구, 고객 문의를 분류하는 도구는 각각 다른 업무 목적과 권한 경계를 가져야 합니다. 좋은 자동화는 “할 수 있는 모든 API”를 여는 것이 아니라, 특정 업무를 안전하게 끝내기 위한 좁고 명확한 도구를 만드는 데서 시작됩니다.

현업도 이 변화에서 빠질 수 없습니다. 에이전트가 제대로 일하려면 업무 규칙이 더 명확해야 합니다. 어떤 경우에는 결재가 필요하고, 어떤 경우에는 조회만 허용하며, 어떤 데이터는 요약해도 되지만 외부 전송은 안 되는지 설명해야 합니다. 지금까지 사람의 눈치와 경험으로 흘러가던 규칙이 AI 앞에서는 도구 설명과 정책으로 바뀌어야 합니다. AX는 결국 업무를 자동화하는 프로젝트이면서, 동시에 업무의 경계를 다시 말로 쓰는 프로젝트가 됩니다.

## ▸ 좋은 AI는 창문보다 문에 가깝다

처음의 출입문으로 돌아가 봅니다. 좋은 사무실의 문은 사람을 막기 위해서만 존재하지 않습니다. 들어와도 되는 사람을 빠르게 들여보내고, 들어오면 안 되는 영역을 지키고, 밤에는 기록을 남기고, 비상시에는 경로를 알려줍니다. 문이 없으면 자유로워 보일 수 있지만, 그 공간은 오래 운영되기 어렵습니다.

AI 에이전트도 비슷합니다. 개인이 쓰는 AI는 창문처럼 느껴집니다. 바깥의 지식과 기능이 갑자기 내 책상 위로 들어오는 경험입니다. 그러나 회사가 쓰는 AI는 문에 가까워야 합니다. 누가 들어오고 나가는지 알고, 어느 방까지 갈 수 있는지 정하고, 움직임을 기록하고, 필요하면 멈출 수 있어야 합니다.

AWS의 Claude apps gateway와 MCP 툴 설계 글이 보여주는 것은 에이전트 시대의 화려한 미래라기보다, 그 미래를 회사 안에 들이기 위한 지극히 실무적인 조건입니다. 모델은 점점 더 강해질 것입니다. 하지만 대기업의 AX를 실제로 가르는 것은 모델의 지능만이 아닙니다. 그 지능이 회사의 문을 통과할 때, 우리는 어떤 권한과 언어와 기록을 함께 쥐여줄 것인가입니다.

앞으로의 질문은 “어떤 AI 앱을 쓸 것인가”에서 끝나지 않습니다. 우리 회사는 AI 에이전트가 안전하게 드나들 수 있는 문을 이미 가지고 있는가, 아니면 아직 창문만 열어두고 있는가.
